# Cadence Virtuoso/OCEAN Parallel Simulation Runbook

This note summarizes how to run a parameter-sweep OCEAN script in parallel from the Cadence CIW using `ipcBeginProcess(...)`, based on the debugging session.

The working approach was not to run multiple jobs inside one CIW session with `load(...)`, but to use the CIW to launch several external OCEAN processes. Each external process receives a different `CAD_JOB_INDEX`, so each one runs a different subset of the simulation sweep.

---

## 1. Core idea

The modified OCEAN script reads two environment variables:

```lisp
CAD_NUM_JOBS
CAD_JOB_INDEX
```

These determine which part of the full sweep a particular OCEAN process runs.

For example, with:

```text
CAD_NUM_JOBS=4
```

there are four logical jobs:

```text
CAD_JOB_INDEX=0
CAD_JOB_INDEX=1
CAD_JOB_INDEX=2
CAD_JOB_INDEX=3
```

If the sweep has 64 valid cases, then each job runs roughly 16 cases:

```text
job 0 → 16 cases
job 1 → 16 cases
job 2 → 16 cases
job 3 → 16 cases
```

The cases reported as “skipped” by one job are not errors. They are the cases assigned to the other job indices.

## 2. What did not work well

### Loading the script directly in CIW

This works for a single sequential job:

```lisp
load("/path/to/single_pwl_opt_parallel.ocn")
```

However, this runs inside the active Virtuoso/CIW process. The CIW becomes busy until the script finishes, so it is not suitable for launching multiple simultaneous jobs.

### `system(...)` commands

Commands such as:

```lisp
system("which ocean")
```

often returned only:

```text
0
```

This is just the shell exit status. On this university Cadence setup, the command output was not reliably displayed in the CIW. This made `system(...)` inconvenient for debugging.

## 3. What worked

The working mechanism was:

```lisp
ipcBeginProcess("...")
```

IPC stands for **Inter-Process Communication**. In this context, `ipcBeginProcess(...)` allows the running Virtuoso/CIW process to launch an external operating-system process, such as `ocean`, without blocking the CIW.

A successful IPC call returned values like:

```text
ipc:13
ipc:15
```

These are process handles, not error codes. They indicate that Cadence successfully started an external process.

For debugging, the explicit four-command version is easier to reason about. For routine use, the one-line `foreach(...)` version is convenient.

## 7. Adapting this to a new directory or larger simulation

For future simulations, change these three things.

### A. OCEAN script path

This is the file passed after `-restore`:

```text
/path/to/your_script.ocn
```

Example:

```text
/home/s5117909/Documents/thesis/sebastian_thesis_repo/processing/general_code/single_pwl_opt_parallel.ocn
```

### B. Log directory

This is where the external OCEAN process writes its terminal output:

```text
/path/to/log_directory/ocean_ipc_job0.log
```

Example:

```text
/home/s5117909/Documents/thesis/sebastian_thesis_repo/processing/cadence_extraction/ocean_ipc_job0.log
```

### C. Output data directory inside the OCEAN script

The actual simulation data are not controlled by the IPC command. They are controlled inside the `.ocn` file, typically by a line like:

```lisp
outputBaseDir = strcat(baseRepoDir "/processing/cadence_extraction/output_single_data")
```

or a similar variable assignment.

For a new simulation campaign, edit the OCEAN script so that `outputBaseDir` points to the desired output directory.

## 10. If paths contain spaces or special characters

Avoid spaces, parentheses, and unusual characters in Cadence/OCEAN working paths if possible. They can make debugging unnecessarily difficult.

## 16. Useful error messages and interpretation

### `Parallel split: job 0 of 1`

The script ran, but no environment variables were passed. It is running sequentially.

Fix: launch with:

```text
CAD_NUM_JOBS=4 CAD_JOB_INDEX=0
```

or use the IPC command form.

### `restore <...> cannot access file for reading`

The OCEAN process started, but the path after `-restore` is wrong or unreadable.

Fix: use the exact absolute path to the `.ocn` file.

### `ocean: command not found`

The IPC process started, but the shell could not find the `ocean` executable.

Fix: use the full path to `ocean` or load the correct Cadence environment before launching.

### `license checkout failed` or similar

The jobs launched, but there may not be enough Cadence/Spectre licenses.

Fix: reduce `CAD_NUM_JOBS`, or wait until licenses are available.

### `WARNING (PRINT-1048)`

The waveform data contain more than 10,000 points, so printing/exporting may be slow.

This is not a parallelisation error. The simulation can still complete successfully.

## 17. Choosing the number of jobs

The total computational load is approximately:

```text
CAD_NUM_JOBS × Spectre threads per job
```

The script currently contains a Spectre option like:

```lisp
envOption('userCmdLineOption "+aps +mt=2")
```

So with:

```text
CAD_NUM_JOBS=4
```

and:

```text
+mt=2
```

the approximate total CPU-thread request is:

```text
4 jobs × 2 threads/job = 8 threads
```

Use a conservative number first. More jobs can overload CPU, memory, disk I/O, or available Cadence licenses.

## 20. Conceptual summary

- `load(...)` runs the OCEAN script inside the current CIW and blocks the CIW.
- `system(...)` was awkward because it mostly returned numeric shell statuses in CIW.
- `ipcBeginProcess(...)` successfully launched external processes and returned handles like `ipc:15`.
- The external process was `ocean -nograph -restore script.ocn`.
- `CAD_NUM_JOBS` and `CAD_JOB_INDEX` tell each process which subset of the sweep to run.
- Job-skipping messages are expected and indicate that the splitting logic is working.
- Output data are written wherever the OCEAN script's `outputBaseDir` points.
- Logs are written wherever the IPC command redirects output with `> logfile 2>&1`.
